3.7. WebAPI e endpoints

Documentacao navegavel da WebAPI do Bull Pace. Esta pagina descreve os endpoints implementados no backend atual, seus contratos de entrada e saida, status codes possiveis e a vinculacao com os Requisitos Funcionais do sistema.

Base local http://localhost:3000
Stack Node.js, Express, TypeScript, EJS e Supabase
Formatos aceitos JSON e formulario HTML
Autenticacao Login visual; sem token JWT validado pelo backend atual

Escopo da documentacao

O servidor atual possui rotas de interface que renderizam HTML e uma API generica em /api/:tabela para consulta e criacao de registros no Supabase. Os arquivos de controller, service, repository e model tambem documentam contratos internos planejados, mas apenas as rotas registradas em src/server.ts estao expostas via HTTP neste momento.

Nesta pagina, o termo Promotor representa o usuario operacional que registra turnos e checkpoints. O termo Coordenadora representa o perfil administrativo que acompanha, audita e finaliza a prova. Alguns nomes de rotas e campos ainda usam operador e coordenador porque esses sao os identificadores implementados no codigo atual.

Status codes como 204, 401, 403, 409 e 422 sao listados como previstos para versoes futuras quando dependem de autenticacao, autorizacao ou validacoes ainda nao implementadas explicitamente no backend atual.

Implementado x Planejado

IMPLEMENTADO API REST generica

GET /api/:tabela e POST /api/:tabela, ambos registrados em src/server.ts e conectados ao Supabase.

IMPLEMENTADO Rotas web HTML

Fluxos de selecao de funcao, Promotor, equipe, atleta, esteira, turno, checkpoints visuais, painel da Coordenadora e Modo TV.

PLANEJADO Endpoints REST especificos

Contratos como /api/turnos/:id/finalizar, /api/checkpoints/turno/:idTurno e /api/placar/:idEvento.

PLANEJADO Autenticacao e autorizacao

Uso real de Authorization, retorno de 401 e retorno de 403 por permissao de perfil.

Arquitetura da WebAPI

Frontend iPad, navegador e telas HTML/EJS.
Controller Recebe req/res e devolve status HTTP.
Service Aplica regras de negocio e validacoes.
Repository Isola consultas e persistencia.
Supabase Banco PostgreSQL usado pela aplicacao.

A API generica atual acessa o Supabase diretamente em src/server.ts. Os contratos planejados indicam a evolucao esperada para o fluxo completo Frontend → Controller → Service → Repository → Supabase.

API REST Implementada

GET /api/:tabela RF001, RF002, RF003, RF007, RF011, RF014, RF015, RF016, RF017
Descricao Consulta todos os registros de uma tabela do Supabase informada na URL.
Parametros de entrada tabela no path. Exemplos: atletas, eventos, esteiras, turnos, checkpoints, sessoes_operacionais.
Headers relevantes Accept: application/json
Body Nao possui body.
Formato de resposta 200 
[
  {
    "id_atleta": 1,
    "id_equipe": 1,
    "nome": "Rafael Luz",
    "status": "ativo"
  }
]
Status codes possiveis
200 consulta realizada 400 tabela invalida, se validacao for adicionada 401 futuro: autenticacao ausente 403 futuro: permissao insuficiente 404 rota inexistente ou tabela nao encontrada 500 erro interno/Supabase
Vinculacao ao RF Apoia consultas de entidades usadas em selecao de equipe, atleta, esteira, checkpoints, placar, resultado e exportacao.
POST /api/:tabela RF005, RF006, RF007, RF009
Descricao Insere um novo registro na tabela do Supabase informada na URL.
Parametros de entrada tabela no path. O corpo deve seguir as colunas da tabela escolhida.
Headers relevantes Content-Type: application/json e Accept: application/json.
Body exemplo 
{
  "id_equipe": 1,
  "nome": "Rafael Luz",
  "status": "ativo"
}
Formato de resposta 201 
[
  {
    "id_atleta": 1,
    "id_equipe": 1,
    "nome": "Rafael Luz",
    "status": "ativo"
  }
]
Status codes possiveis
201 criado 400 body malformado 401 futuro: autenticacao ausente 403 futuro: permissao insuficiente 404 tabela nao encontrada 409 conflito de regra de negocio 422 dados invalidos para constraints 500 erro interno/Supabase
Vinculacao ao RF Apoia criacao de registros operacionais como turnos, checkpoints e timestamps automaticos.
